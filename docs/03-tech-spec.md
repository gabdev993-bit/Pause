## 1. 전체 아키텍처 개요 (2026-07-15 업데이트: 두 기기 구조로 확정)

기존에는 한 대의 공유 마이크로 두 사람을 모두 잡는 구조를 v1으로 잡았으나, "동시 발화(겹치는 말) 처리"라는 어려운 기술 문제를 프로토타입 단계에서 정면 돌파하는 대신, **각자 폰으로 세션에 참여하는 두 기기 구조를 v1부터 채택**하기로 결정함. 각 기기가 자신의 소유자 목소리를 상대적으로 선명하게 포착하기 때문에, 화자 분리를 서버가 애써 계산할 필요가 크게 줄어든다.

```
[기기 A - 브라우저]              [기기 B - 브라우저]
   ├─ getUserMedia                  ├─ getUserMedia
   ├─ 로컬 음향 분석                 ├─ 로컬 음향 분석
   │   (음량/발화속도)                │   (음량/발화속도)
   └─ WebSocket ─┐              ┌─ WebSocket
                 │              │
                 ▼              ▼
        [백엔드 서버 — 세션 룸(Room) 단위로 두 스트림을 페어링]
                 │
   ├─ 두 기기의 로컬 음향 지표를 타임스탬프 기준으로 비교
   │     → "둘 다 동시에 말하고 있음" 여부만 가볍게 판단 (겹침 근사)
   │
   ├─ 각 기기 오디오를 STT 스트리밍 API로 개별 전달 (AssemblyAI/Deepgram)
   │     → 화자 분리 없이도 "이 텍스트는 A의 말"이 자동으로 보장됨
   │
   ├─ 텍스트 청크 수신 시 LLM 호출 (비난조 감지 + 리프레이밍 제안)
   └─ 결과를 두 기기 모두에 WebSocket으로 실시간 반환 (동일 내용, 동일 UI)

[기기 A/B 공통]
   └─ 넛지 UI 표시 (로컬 음향 기반 넛지는 즉시, 의미 기반 제안은 1~2초 지연 후)
```

핵심 설계 원칙: **화자 분리를 알고리즘으로 풀지 않고 기기 배정으로 해결한다.** "누가 말했는지"는 어느 기기에서 들어온 오디오인지로 자동 결정되므로, 서버는 두 텍스트 스트림을 시간순으로 병합하기만 하면 된다. 대신 두 기기를 동시에 세션에 붙이는 문제(페어링·세션 참여 플로우)가 새로운 핵심 과제가 된다.

## 2. 세션 페어링·참여 시그널링 (두 기기 구조의 핵심 신규 과제)

- **최초 페어링(1회)**: A가 초대 링크 생성 → B가 링크 클릭 → 두 계정이 서버에 "커플" 단위로 저장됨 (PRD 2.1 참고)
- **세션 시작**: A가 "시작" 탭 → 백엔드가 세션 ID(Room)를 발급하고 WebSocket 연결 대기 상태로 둠
- **세션 참여**: B가 자기 폰에서 앱을 열고 "참여" 탭 → 이미 페어링된 상대(A)의 진행 중인 세션이 있으면 자동으로 같은 Room에 연결
- **v1 범위**: 푸시 알림으로 B의 기기를 자동으로 깨우지 않음 (iOS가 백그라운드 마이크 접근을 차단하므로 실효성 없음, PRD "제외" 목록 참고). B는 A가 말로 알리는 것을 듣고 직접 앱을 열어 참여
- **연결 실패 처리**: B가 일정 시간(예: 60초) 내 참여하지 않으면 A의 화면에 "아직 상대가 참여하지 않았어요" 표시, A 혼자서도 세션을 계속할지 선택 가능(둘 다 필요하다는 원칙은 유지하되, 강제로 막지는 않음)

## 3. 1단계: 로컬 음향 레벨 격화 감지 (기기별)

- **입력**: 각 기기의 `getUserMedia` raw PCM 스트림 (자기 소유자 목소리가 상대적으로 선명함)
- **처리**: `AudioWorklet`에서 프레임 단위(예: 20ms)로 다음을 계산
  - RMS 음량(dB) — 절대값 및 직전 대비 변화율
  - Zero-crossing rate 등을 이용한 대략적 발화 속도 추정
  - 자신의 음성 활동 여부(VAD, Voice Activity Detection) — "지금 이 사람이 말하고 있다/아니다"만 판단
- **서버 병합**: 두 기기에서 온 VAD 신호를 타임스탬프로 비교해 "두 사람이 동시에 말하고 있음" 구간을 근사 — 이제는 한 마이크에서 화자를 분리하는 어려운 문제가 아니라, 두 개의 단순 on/off 신호를 겹쳐보는 훨씬 쉬운 문제가 됨
- **출력**: 음량·속도·동시발화 여부를 슬라이딩 윈도우(예: 최근 10초)로 집계해 "격화 스코어" 산출
- **트리거**: 격화 스코어가 임계치를 넘으면 즉시 양쪽 UI에 넛지 표시 (텍스트 분석 결과를 기다리지 않음)

## 4. 2단계: 의미 레벨 분석 (기기별 스트림, 겹침 대기 불필요)

- **선행 조건 완화**: 이전 설계(한 기기 공유 마이크)에서는 "겹치지 않는 구간"만 기다려야 했으나, 두 기기 구조에서는 각자 스트림이 이미 분리되어 있으므로 이 제약이 사실상 사라짐 — 상시로 STT 처리 가능
- **STT**: 각 기기의 오디오 청크를 WebSocket으로 백엔드에 전달 → 백엔드가 스트리밍 STT API(AssemblyAI 또는 Deepgram)로 텍스트 변환 (어느 기기에서 왔는지가 곧 "누가 말했는지")
  - 대안: 브라우저 내장 Web Speech API 사용 시 서버 왕복 없이 클라이언트에서 바로 텍스트 획득 가능 (단, Safari 지원 편차 고려해 폴백 로직 필요)
- **분류/제안**: 변환된 문장을 LLM에 프롬프트로 전달 → "비난/경멸/방어/담쌓기(Gottman Four Horsemen)" 패턴 여부 판단 및 리프레이밍 문장 제안
- **출력**: 판단 결과가 있을 때만 양쪽 클라이언트에 카드로 표시 (강제 팝업 아님, 참고용)

## 5. 기술 스택 제안 (프로토타입 기준)

| 영역 | 선택지 | 비고 |
|---|---|---|
| 프론트엔드 | Next.js (React) | 빠른 프로토타이핑, Vercel 등으로 즉시 배포 가능 |
| 오디오 캡처/분석 | Web Audio API, AudioWorklet | 1단계 로컬 분석용 |
| 브라우저 내장 STT | Web Speech API (Chrome/Edge 우선, Safari 폴백) | 비용 없음, 프로토타입 초기엔 이걸로 시작 가능 |
| 클라우드 스트리밍 STT | AssemblyAI Realtime ($0.15/hr) 또는 Deepgram | 정확도·안정성이 더 필요해지면 전환 |
| LLM (분류·리프레이징) | Claude/GPT 계열 소형 모델 API | 응답 속도 우선, 프롬프트는 Gottman 이론 기반으로 설계 |
| 백엔드 | Node.js (Fastify/Express) 또는 Python (FastAPI) | WebSocket 브리지 역할만 하는 경량 서버 |
| 배포 | Vercel(프론트) + Fly.io/Render(백엔드) | 초기 비용 최소화 |

## 6. 데이터·프라이버시 처리

- 세션 시작 전 양쪽 모두 명시적 동의 화면 (PRD 6절 플로우 참고)
- 원본 오디오는 기본적으로 **서버에 영구 저장하지 않음** — STT 처리 후 즉시 폐기
- 텍스트 트랜스크립트·리프레이징 제안 로그는 세션 단위로만 임시 보관, 세션 종료 시 사용자가 저장/삭제 선택
- 세션 진행 중에는 브라우저 화면에 "녹음 중" 표시를 상시 노출 (은밀한 감청처럼 보이지 않도록)

## 7. 알려진 기술적 한계 (리서치 문서 기반)

1. **동시 발화 처리는 완전히 해결된 게 아니라 완화된 것**: 두 기기 구조로 "정밀한 화자 분리"는 피했지만, 두 사람이 물리적으로 아주 가까이 붙어 있으면 서로의 목소리가 상대 기기 마이크에도 섞여 들어감 — 완벽한 격리는 아님. 초기 캘리브레이션으로 보정 필요
2. **기기 간 지연·동기화**: 두 스트림의 타임스탬프가 정확히 맞아야 "동시에 말했다"를 판단할 수 있음 — 네트워크 지연이 다르면 오판 가능. NTP 기반 시간 동기화 또는 서버 수신 시각 기준 근사가 필요
3. **연결 안정성**: 두 기기 중 하나라도 네트워크가 끊기면 세션이 절반만 작동 — 재연결 로직과 "상대 연결 끊김" 안내가 필요 (섹션 2 "연결 실패 처리" 참고)
4. **배터리·발열**: 두 기기가 동시에 마이크+네트워크를 계속 사용하므로 긴 세션일수록 배터리 소모가 큼 — 프로토타입 단계에서는 세션이 대체로 짧을 것으로 가정하고 우선순위 낮게 둠
5. **iOS Safari의 Web Speech API 제약**: 프리픽스 필요(`webkitSpeechRecognition`), 네트워크 연결 필수, 일부 기기에서 "받아쓰기" 설정 활성화 필요 → 폴백으로 클라우드 STT 경로 준비
6. **모바일 브라우저 백그라운드 제약**: 탭이 백그라운드로 가면 오디오 처리 중단 가능성 높음 → v1은 "세션 진행 중 화면을 보고 있는" 포그라운드 사용을 전제로 설계, 이 때문에 푸시 알림으로 상대를 자동 참여시키는 것도 불가능함(섹션 2 참고)

## 8. 향후 단계 (Post-MVP)

- 두 기기 지연 동기화 정밀도 개선, 근접 시 마이크 크로스토크(상대 목소리가 섞여 들어오는 것) 보정
- 온디바이스 STT/LLM 경량화로 서버 의존도 축소 (프라이버시 강화)
- 네이티브 앱 전환 시 백그라운드/알림 기반 개입(상대 기기를 실제로 깨워서 자동 참여) 추가
- 위기 신호(가정폭력 등) 감지 시 전문기관 연결 플로우 설계 (별도 문서 필요)
