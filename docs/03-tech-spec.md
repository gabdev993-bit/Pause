## 1. 아키텍처 개요

Pause는 외부 통화나 메신저를 분석하는 앱이 아니라, 페어링된 커플이 텍스트와 음성통화를 직접 사용하는 하나의 커뮤니케이션 공간이다.

```
[기기 A 브라우저]                         [기기 B 브라우저]
  ├─ 지속 대화 공간                         ├─ 지속 대화 공간
  ├─ 텍스트 입력·개인 코칭                  ├─ 텍스트 입력·개인 코칭
  ├─ WebRTC 음성 스트림                     ├─ WebRTC 음성 스트림
  └─ 로컬 음향 지표 ─────┐       ┌──────── 로컬 음향 지표
                         ▼       ▼
                    [백엔드 / 세션 서버]
                    ├─ 커플·대화·통화 상태 관리
                    ├─ 텍스트 저장·전송
                    ├─ 음향 지표 병합
                    ├─ STT·의미 분석
                    └─ 공유 넛지 / 개인 코칭 반환
```

핵심 원칙은 분석 결과를 판정이나 점수로 저장하지 않고, 대화 중 필요한 순간에만 최소한으로 개입하는 것이다.

## 2. 커플 및 대화 공간

- 최초 1회: A가 초대 링크를 생성하고 B가 링크를 열어 연결에 동의한다.
- 연결 완료 후 서버에 커플 관계와 지속적인 1:1 대화 공간을 생성한다.
- 이후 대화마다 다시 페어링하거나 세션을 새로 만들지 않는다.
- MVP에서는 텍스트 메시지와 Pause 내부 음성통화만 제공한다.
- 그룹 채팅, 영상통화, 파일·스티커 전송은 제공하지 않는다.

## 3. 텍스트 메시징

- 일반적인 1:1 메시지 전송 구조를 사용한다.
- 전송 전 초안은 작성자 기기에서 분석하거나 임시 서버 처리 후 즉시 폐기한다.
- 다음 신호를 보수적으로 감지한다: 관계 위협, 모욕·경멸, 반복 비난, 짧은 시간 내 연속 전송, 과도한 대문자·구두점, 동일 요구 반복.
- 높은 확신의 패턴일 때만 작성자에게 개인 코칭을 표시한다.
- 작성자는 원문 전송, 직접 수정, 제안 표현 사용 중 하나를 선택한다.
- 개인 코칭과 원래 초안은 상대에게 공개하지 않는다.
- 메시지 저장 범위와 삭제·보존 기간은 별도 정책으로 확정한다.

## 4. Pause 내부 음성통화

- 통화 요청은 커플 대화 공간에서 시작한다.
- B가 수락해야 WebRTC 통화가 연결된다. 거절·부재·취소 상태를 양쪽에 표시한다.
- 양쪽 모두 마이크 권한과 AI 분석에 명시적으로 동의해야 분석을 시작한다.
- WebRTC로 음성 스트림을 연결하고, 각 기기의 입력 스트림을 A/B로 구분한다.
- 연결 상태, 재연결 중, 상대 연결 끊김, 통화 종료를 화면에 명확히 표시한다.
- 통화 종료 시 마이크와 분석을 즉시 중단한다.

## 5. 실시간 분석과 개입

### 5.1 음향 레벨

각 기기에서 음량, 발화 속도, 음성 활동을 로컬 계산하고 서버에서 시간창 단위로 결합한다.

- 개인 기준 대비 음량 상승
- 발화 속도 증가
- 반복적인 동시 발화·끼어들기
- 네트워크 지연으로 인한 불확실 상태

높은 확신으로 대화가 격화되었다고 판단할 때 양쪽에 공유 넛지를 표시한다. 누구의 책임인지 나타내지 않는다.

### 5.2 의미 레벨

- 음성 스트림은 스트리밍 STT로 변환한다.
- 텍스트와 음성 모두 동일한 중립성 원칙으로 분류한다.
- 관계 위협, 비난, 경멸, 방어, 담쌓기 표현을 감지한다.
- 공유 넛지는 양쪽에 동일하게, 개인 코칭은 해당 작성자·발화자에게만 표시한다.
- 개입은 화면 카드 등 비강제 방식으로 제공하며 자동 전송·자동 종료는 하지 않는다.

## 6. 데이터 및 프라이버시

- 원본 음성은 실시간 처리 후 영구 저장하지 않는다.
- 상대 동의 없이 통화 분석을 시작하지 않는다.
- AI 중간 데이터의 보존 시간을 최소화한다.
- 통화 종료 직후 마이크·분석 중단 상태를 명시적으로 표시한다.
- 사용자 대화를 모델 학습에 사용하려면 별도 명시 동의를 받는다.
- 메시지 보존·삭제 정책은 개발 착수 전에 확정하고 실제 저장소 삭제까지 검증한다.

## 7. 권장 기술 스택

| 영역 | 제안 |
|---|---|
| 프론트엔드 | Next.js / React |
| 실시간 대화 | WebSocket |
| 음성통화 | WebRTC + STUN/TURN |
| 오디오 분석 | Web Audio API, AudioWorklet |
| STT | Web Speech API 초기 검증, 필요 시 AssemblyAI/Deepgram |
| 의미 분류 | 소형 LLM API, 중립성·안전 프롬프트 |
| 백엔드 | Node.js(Fastify/Express) 또는 FastAPI |
| 저장소 | 암호화된 관계·메시지 저장소, 임시 분석 데이터 TTL |
| 배포 | Vercel + WebSocket/WebRTC 지원 백엔드 |

## 8. 알려진 한계

- 모바일 브라우저 백그라운드에서는 통화·분석이 중단될 수 있다.
- 네트워크 지연과 마이크 크로스토크가 음향 분석 정확도에 영향을 준다.
- STT가 감정과 문맥을 잘못 이해할 수 있으므로 높은 확신에만 개입한다.
- 학대·강압 관계에서는 일반 중재가 위험을 높일 수 있으므로 안전 분기 정책이 필요하다.
- 통화 요약·감정 그래프는 핵심 가설 검증 이후로 미룬다.
