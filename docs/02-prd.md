# PRD — Pause

작성일: 2026-07-17
버전: v0.2 (제품 방향 재정의)
근거 문서: `01-research-summary.md`

---

## 1. 한 줄 정의

Pause는 연인들이 어렵거나 감정적인 대화를 내부 음성통화와 텍스트로 나누면서, 대화가 서로를 해치는 방향으로 흐르기 전에 잠시 늦춰주는 커뮤니케이션 공간이다.

## 2. 문제 정의

연인 간 갈등은 대화의 주제 자체보다 대화가 진행되는 방식 때문에 더 크게 악화되기도 한다.

특히 장거리 연애처럼 관계의 상당 부분이 통화와 텍스트로 이루어지는 경우 다음 문제가 반복된다.

* 통화 중 목소리가 커지거나 서로의 말을 끊는다.
* 짧은 텍스트를 실제 의도보다 부정적으로 해석한다.
* 답장이 늦으면 불안해져 메시지를 연속으로 보낸다.
* 감정적으로 작성한 메시지를 곧바로 전송한다.
* "헤어지자", "됐어", "마음대로 해" 같은 관계 위협 표현이 나온다.
* 한 사람이 전화를 끊거나 답장을 중단하면서 불안과 분노가 더 커진다.
* 싸움이 끝난 뒤에는 이미 상처를 준 말을 되돌리기 어렵다.

기존 통화·메신저 앱은 메시지를 빠르게 전달하는 데 최적화되어 있을 뿐, 대화가 감정적으로 안전하게 진행되도록 돕지 않는다.

반면 기존 관계 앱은 대부분 평온할 때 사용하는 질문, 퀴즈, 저널 중심이어서 실제 갈등 대화 안에는 존재하지 않는다.

## 3. 핵심 제품 가설

> 커플이 어렵거나 민감한 대화를 처음부터 Pause 안에서 나눈다면, 갈등 도중 별도의 앱을 새로 켜지 않아도 적절한 순간에 개입할 수 있다.

Pause는 외부에서 진행 중인 대화를 감청하거나 분석하는 도구가 아니다. Pause 자체가 두 사람이 통화하고 텍스트를 주고받는 장소다.

v1에서 검증할 가설:

1. 커플은 민감한 대화를 위해 별도의 안전한 공간을 사용할 의향이 있는가?
2. 중립적인 실시간 개입이 대화를 실제로 늦추거나 완화하는가?
3. 개입이 감시나 판정이 아니라 도움으로 느껴지는가?
4. 음성과 텍스트에서 같은 중재 원칙이 작동하는가?
5. 두 사람 모두 참여해야 하는 구조가 실제 사용에서 성립하는가?

## 4. 타겟 사용자

### 4.1 1차 타겟

장거리 연애 중이며 통화나 텍스트에서 반복적으로 갈등을 겪는 커플.

특히 다음 조건에 해당하는 커플을 우선한다.

* 두 사람 모두 관계를 개선하려는 의지가 있다.
* 중요한 대화를 주로 음성통화나 텍스트로 나눈다.
* 통화 중 끼어들기, 언성 상승, 갑작스러운 종료가 반복된다.
* 텍스트의 말투나 답장 속도 때문에 오해가 자주 생긴다.
* 싸운 뒤 "그 말을 보내지 말았어야 했다"고 후회한다.

### 4.2 2차 타겟

같은 지역에 살더라도 민감한 이야기를 전화나 텍스트로 자주 나누는 커플.

장거리 연애는 초기 진입 시장이지만, Pause를 장거리 전용 제품으로 제한하지 않는다.

### 4.3 명시적 비대상

* 가정폭력, 강압적 통제 또는 즉각적인 안전 위험이 있는 관계
* 상대의 동의 없이 대화를 녹음하거나 분석하려는 사용자
* 관계 개선 의사가 없는 상대를 설득하거나 감시하려는 사용자
* Pause를 치료나 전문 상담의 대체재로 사용하려는 사용자

위험 신호가 발견되면 일반적인 대화 중재를 계속하기보다 안전 안내와 전문기관 연결로 전환해야 한다.

## 5. 핵심 사용 상황

Pause는 모든 일상 대화를 대체하는 범용 메신저로 시작하지 않는다. 다음과 같은 대화를 위한 공간이다.

* 반복해서 싸우는 문제를 다시 이야기할 때
* 서운함이나 불안을 설명하고 싶을 때
* 관계의 미래에 관한 중요한 결정을 이야기할 때
* 이미 감정이 올라왔지만 대화를 계속하고 싶을 때
* 텍스트로 작성한 말을 보내기 전에 한 번 점검하고 싶을 때
* 기존 통화나 메신저에서 갈등이 시작되어 더 안전한 공간으로 옮기고 싶을 때

Pause가 만들고자 하는 핵심 행동은 다음과 같다.

> "이 이야기는 Pause에서 하자."

## 6. 핵심 가치 제안

| 기존 통화·메신저          | Pause                       |
| ------------------ | --------------------------- |
| 대화를 빠르게 전달하는 데 집중  | 필요한 순간 반응 속도를 늦추고 생각할 틈을 제공 |
| 갈등 표현을 그대로 전달      | 갈등이 커지는 패턴을 감지해 중립적으로 개입    |
| 감정적인 메시지도 즉시 전송    | 보내기 전 표현을 돌아보고 수정할 기회 제공    |
| 통화 종료가 곧 단절이 되기 쉬움 | 멈춤을 관계 포기가 아닌 대화 조절 행동으로 전환 |
| 갈등이 끝난 후 기록만 남음    | 갈등이 진행되는 순간에 도움 제공          |
| 개별 사용자 단위 과금       | 1구독으로 연결된 커플 두 명이 함께 사용     |

## 7. 제품 설계 원칙

### 7.1 중립성

Pause는 누구의 말이 옳은지 판단하지 않는다.

* 누가 싸움을 시작했는지 평가하지 않는다.
* 한 사람에게 책임 점수나 공격성 점수를 부여하지 않는다.
* "당신이 문제입니다"처럼 개인을 지목하지 않는다.
* 한쪽의 주장만으로 관계나 상대를 진단하지 않는다.

### 7.2 공유 넛지와 개인 코칭의 분리

모든 분석 결과를 양쪽에 똑같이 공개하면 한 사람이 지적받거나 수치심을 느낄 수 있다. 따라서 개입을 두 종류로 구분한다.

#### 공유 넛지

양쪽이 함께 보는 중립적인 메시지다.

* "두 분의 대화 속도가 빨라지고 있어요."
* "잠깐 속도를 늦춰볼까요?"
* "서로의 말을 끝까지 들을 시간이 필요해 보여요."

#### 개인 코칭

작성자나 발화자에게만 보이는 도움이다.

* 메시지를 보내기 전 다시 읽어보기
* 비난 표현을 감정과 욕구 표현으로 바꾸기
* 잠시 기다린 뒤 보내기
* 말하고 싶은 핵심을 짧게 정리하기

상대에게 사용자의 원래 초안, 개인 분석 결과 또는 개인 코칭 내용을 공개하지 않는다.

### 7.3 강제하지 않기

* 메시지 전송을 강제로 막지 않는다.
* 통화를 자동으로 종료하지 않는다.
* AI가 사용자를 대신해 말하거나 메시지를 자동 전송하지 않는다.
* 모든 제안은 무시하거나 닫을 수 있다.

### 7.4 갈등을 점수화하지 않기

누가 더 화냈는지, 누가 더 공격적이었는지, 누가 갈등의 원인인지 결과로 보여주지 않는다. 분석 결과가 다음 싸움의 증거로 사용되지 않도록 한다.

### 7.5 극단적 단순함

사용자에게 먼저 "통화 모드"와 "텍스트 모드" 중 하나를 고르게 하는 설정 화면을 만들지 않는다.

페어링된 상대와의 하나의 공간 안에 항상 다음 두 가지가 존재한다.

* 텍스트 입력창
* 음성통화 버튼

세션 시간, 코칭 스타일, 대화 유형 같은 사전 설정은 요구하지 않는다.

## 8. 핵심 경험

### 8.1 텍스트 대화

1. 사용자가 페어링된 상대와의 대화방을 연다.
2. 메시지를 작성한다.
3. 위험도가 높은 표현이 감지되면 전송 전에 개인 코칭을 보여준다.
4. 사용자는 원문 전송, 직접 수정 또는 제안 표현 사용을 선택한다.
5. 대화 전체가 빠르게 격화되면 양쪽에 공유 넛지를 표시한다.
6. 사용자는 계속 대화하거나 잠시 멈출 수 있다.

초기 감지 신호:

* 관계 위협 표현
* 모욕·경멸
* 반복적인 비난
* 짧은 시간 안의 연속 메시지
* 상대 응답 전 반복 전송
* 과도한 대문자·구두점
* 같은 요구나 질문의 반복

### 8.2 내부 음성통화

1. 사용자가 대화방의 통화 버튼을 누른다.
2. 상대에게 Pause 통화 요청이 전달된다.
3. 상대가 수락하면 Pause 내부에서 음성통화가 시작된다.
4. 두 사람의 음성 스트림을 기기별로 분리해 분석한다.
5. 언성 상승, 발화 속도 증가, 반복적인 끼어들기 또는 위험한 표현을 감지한다.
6. 양쪽에는 공유 넛지를, 필요한 개인에게는 개인 코칭을 보여준다.
7. 누구든 통화를 종료할 수 있다.

초기 감지 신호:

* 개인 기준 대비 음량 상승
* 발화 속도 증가
* 반복적인 동시 발화와 끼어들기
* 비난·경멸·방어·담쌓기 표현
* 관계 위협 표현
* 반복적인 통화 종료 시도

## 9. MVP 범위

### 9.1 P0 — 반드시 포함

* 최초 1회 파트너 페어링
* 연결된 커플을 위한 하나의 지속적인 대화 공간
* 1:1 텍스트 메시지
* Pause 내부 1:1 음성통화
* 통화 요청, 수락·거절, 연결 상태 표시
* 마이크 권한 및 AI 분석에 대한 양쪽의 명시적 동의
* 음성 스트림의 기기별 분리
* 음량·발화 속도·동시 발화 감지
* 음성 의미 분석
* 텍스트 초안 분석
* 양쪽이 함께 보는 중립적 공유 넛지
* 개인에게만 보이는 리프레이밍 제안
* 통화 종료
* 연결 실패 및 재연결 안내
* 원본 음성 비영구 저장
* 최소한의 신고·차단·안전 안내 구조

### 9.2 P1 — MVP 이후

* 통화 요약
* 감정 온도 그래프
* 통화 중 표시되었던 제안 목록
* "10분 후 다시 이야기하기" 같은 합의형 타임아웃
* 통화에서 텍스트로 자연스럽게 이어가기
* 개입이 도움이 되었는지에 대한 간단한 피드백
* 위기 신호 감지 및 지역별 전문기관 안내 고도화
* 메시지 삭제와 세부 데이터 관리
* 민감한 대화 시작 링크
* 세션 히스토리

통화 요약과 감정 그래프는 실시간 개입의 핵심 가설이 검증된 후에 개발한다. 특히 감정 그래프가 상대를 평가하거나 다음 갈등의 증거로 사용되지 않는지 먼저 확인해야 한다.

### 9.3 명시적 비목표

* 모든 일상 대화를 대체하는 범용 메신저
* 그룹 채팅과 그룹 통화
* 영상통화
* 사진·파일·스티커 전송
* 데일리 퀴즈와 무드 체크인
* 개인 저널
* 관계 점수와 게임화
* 누가 더 잘못했는지 분석
* 상대방 감시
* 상대 동의 없는 자동 녹음
* AI 연인 또는 AI 상담사
* 커뮤니티 기능
* 생체 데이터 연동

## 10. 핵심 사용자 플로우

### 10.1 최초 연결

1. A가 Pause에서 파트너 초대 링크를 생성한다.
2. B가 링크를 열고 연결에 동의한다.
3. 두 사용자에게 하나의 지속적인 커플 대화 공간이 생성된다.
4. 이후 매 대화마다 다시 페어링하지 않는다.

### 10.2 텍스트 대화

1. 사용자가 커플 대화 공간을 연다.
2. 텍스트를 작성한다.
3. 필요한 경우 전송 전 개인 코칭이 나타난다.
4. 사용자가 원문 또는 수정된 메시지를 전송한다.
5. 대화가 격화되면 양쪽에 공유 넛지가 나타난다.
6. 대화는 별도 완료 절차 없이 이어지거나 멈춘다.

### 10.3 음성통화

1. A가 커플 대화 공간에서 통화 버튼을 누른다.
2. B가 통화 요청을 받고 수락한다.
3. 양쪽의 동의와 마이크 권한을 확인한다.
4. Pause 내부 음성통화와 실시간 분석이 시작된다.
5. 격화 신호가 발견되면 공유 넛지 또는 개인 코칭이 나타난다.
6. 어느 한쪽이 종료하면 마이크와 분석이 즉시 중단된다.
7. v1에서는 별도의 통화 요약이나 감정 그래프 없이 대화 공간으로 돌아간다.

## 11. 데이터·프라이버시 원칙

* 원본 음성은 실시간 처리 후 영구 저장하지 않는다.
* 상대의 동의 없이 통화 분석을 시작할 수 없다.
* 개인 코칭 내용과 전송 전 텍스트 초안은 상대에게 공개하지 않는다.
* AI 분석을 위해 생성된 중간 데이터의 보존 시간을 최소화한다.
* 텍스트 메시지의 저장 범위와 보존 기간은 개발 착수 전에 별도로 확정한다.
* 삭제 기능을 제공할 경우 실제 저장소에서도 데이터가 제거되어야 한다.
* 사용자 대화를 모델 학습에 사용하려면 별도의 명시적 동의를 받아야 한다.
* 통화가 종료되면 마이크와 분석이 즉시 중단되었다는 사실을 명확히 표시한다.

## 12. 성공 지표

### 12.1 핵심 지표

개입이 발생한 대화 중, 개입 이후 일정 시간 안에 격화 신호가 감소한 비율.

예시:

* 음량과 동시 발화가 60초 이내 감소
* 메시지 전송 속도가 느려짐
* 공격적인 텍스트 초안이 전송 전에 수정됨
* 관계 위협 표현이 전송 전에 변경됨

### 12.2 제품 지표

* 파트너 페어링 완료율
* Pause 통화 요청 대비 수락률
* 두 사람이 모두 참여한 대화의 완료율
* 민감한 대화에서의 재사용률
* 텍스트 제안 후 수정률
* 음성 넛지 후 격화 감소율
* 두 사람이 모두 개입이 도움이 되었다고 평가한 비율

### 12.3 보호 지표

* 잘못된 개입으로 느낀 비율
* 한쪽 편을 든다고 느낀 비율
* 감시받는다고 느낀 비율
* 넛지 직후 통화 종료율
* 개인 코칭이 상대에게 노출되었다고 느낀 사례
* 안전 문제 또는 강압적 사용 신고율

## 14. 차별화

* Pause 자체가 내부 음성통화와 텍스트를 제공하므로 갈등 순간에 외부 대화를 몰래 듣거나 별도 감지 앱을 켤 필요가 없다.
* 일반 메신저를 대체하기보다 어렵거나 민감한 대화를 위한 커뮤니케이션 공간에 집중한다.
* 사후 분석보다 대화가 진행되는 순간의 개입에 집중한다.
* 음성과 텍스트를 같은 중립성 원칙으로 지원한다.
* 공유 넛지와 개인 코칭을 분리해 한쪽이 공개적으로 지적받는 경험을 줄인다.
* 퀴즈, 저널, 관계 점수, 세션 시간 설정 없이 실제 대화 자체에만 집중한다.
* 1구독으로 연결된 두 사용자가 함께 이용한다.

## 15. 개발 착수 전 오픈 이슈

1. Pause를 어려운 대화 전용으로만 포지셔닝할지, 이후 일반 커플 통화·메시지로 확장할지
2. 텍스트 메시지를 영구 보관할지, 일정 기간 후 삭제할지
3. 음성통화 중 개인 코칭을 화면, 소리, 진동 중 어떤 방식으로 전달할지
4. 사용자가 통화 중 화면을 보고 있지 않을 때 공유 넛지를 어떻게 전달할지
5. 관계 위협 표현을 어느 수준부터 개입 대상으로 볼지
6. 통화를 갑자기 끊는 행동을 감지하거나 개입 대상으로 볼지
7. AI 분석을 끈 상태에서도 일반 통화와 텍스트를 허용할지
8. 초기 웹 프로토타입에서 통화 요청을 어떤 방식으로 전달할지
9. 텍스트 메시지 보존과 삭제 정책을 어떻게 설계할지
