## 0. MVP 한 줄 목표

**"둘이 다투다가 격해질 때, 버튼 하나로 시작해서 잠깐 멈추게 해주는 도구가 실제로 켜지는가?"를 검증한다.**
기능을 늘리는 게 목표가 아니라, 이 한 줄이 실제로 일어나는지 확인하는 게 목표다. (PRD 4-1절 원칙 적용)

## 2. P0 기능 상세 정의

### 2.1 최초 1회 파트너 페어링

- **트리거**: 앱(웹)을 처음 열었을 때, 파트너와 연결되어 있지 않은 상태
- **동작**: 링크 생성 → 상대에게 공유(문자/카톡 등 외부 채널로) → 상대가 링크 클릭 시 자동으로 페어링 완료
- **완료 기준**: 페어링 완료 후에는 이후 세션에서 이 과정을 다시 요구하지 않음 (재로그인/새 기기 예외 제외)
- **예외 처리**: 링크 만료(예: 24시간), 상대가 아직 안 눌렀을 때의 대기 상태 UI
- **명시적 비목표**: Harmony처럼 매 세션 시작 시 "파트너 연결 대기" 화면을 다시 띄우지 않음

### 2.2 동의·프라이버시 화면

- **트리거**: 각 세션 시작 직전, 페어링된 두 사람 모두에게 노출
- **동작**: "이 세션은 마이크로 대화를 듣고 분석합니다. 원본 음성은 저장하지 않습니다"를 명확히 표시 → 양쪽 모두 동의해야 세션 시작
- **완료 기준**: 동의하지 않은 인원이 있으면 세션이 시작되지 않고 이유를 표시
- **예외 처리**: 동의 화면이 매번 나오면 마찰이 되므로, "이 기기에서 다시 묻지 않기" 옵션 제공 (단, 최초 1회는 반드시 양쪽 다 명시적으로 동의)
- **법적 근거**: 리서치 문서 6절 — 양자 동의(all-party consent) 요건 충족 목적

### 2.3 세션 시작 (두 기기, 2026-07-15 결정 반영)

- **트리거**: A의 메인 화면 단일 버튼("시작") 탭
- **동작**: 백엔드에 세션 Room 생성 → A의 마이크 활성화 → 화면에 "듣고 있음, 상대 참여 대기 중" 표시
- **완료 기준**: 버튼 탭 후 1초 이내 A쪽 마이크 활성화 및 상태 표시 시작
- **명시적 비목표**: 시간 선택, Solo/With Partner 선택, Chat/Voice 선택 없음 — 세션은 항상 "둘이 음성으로"만 존재. 푸시 알림으로 B를 자동 참여시키지 않음 (기술 스펙 2절 참고)

### 2.3-1 세션 참여 (상대 기기, 신규)

- **트리거**: B가 자기 폰에서 앱을 열고 "참여" 탭 (A가 말로 알렸다고 가정)
- **동작**: 페어링된 상대(A)의 진행 중인 세션을 자동 감지 → 같은 Room에 연결 → B의 마이크도 활성화
- **완료 기준**: A와 B 양쪽 화면 모두 "듣고 있음" 상태로 전환, 이 시점부터 세션이 실질적으로 시작됨
- **예외 처리**: B가 60초 내 참여하지 않으면 A 화면에 "아직 참여하지 않았어요" 표시, A는 혼자 계속할지 선택 가능 (기술 스펙 2절 "연결 실패 처리")

### 2.4 1단계: 음향 레벨 격화 감지 (기기별 + 서버 병합)

- **정의**: 각 기기가 로컬에서 STT 없이 자신의 음량·발화속도·음성활동(VAD)을 계산 → 서버가 두 기기 신호를 타임스탬프로 비교해 "격화 스코어" 산출 (기술 스펙 3절 참고, 화자 분리는 기기 배정으로 대체됨)
- **트리거**: 격화 스코어가 사전 정의 임계치 초과
- **동작**: 2.5의 넛지 UI 표시 (양쪽 기기 동시)
- **완료 기준(정확도 목표는 프로토타입 단계이므로 잠정)**: 명백히 격해진 순간(음량 급증 + 양쪽 동시발화 급증)에서 10초 이내 넛지 노출
- **예외 처리**: 배경 소음(TV, 아이 울음 등)으로 인한 오탐 완화를 위해 초기 임계치는 보수적으로 설정 (PRD 8절 리스크 대응). 두 사람이 물리적으로 붙어 있으면 상대 목소리가 내 기기에도 섞여 들어올 수 있음(기술 스펙 7절 한계 1번) — 초기 캘리브레이션 필요

### 2.5 넛지 UI (중립적 알림)

- **정의**: "지금 대화 온도가 올라가고 있어요, 잠깐 멈춰볼까요?" 같은 문구를 화면에 표시
- **원칙**: 항상 양쪽 모두에게 동일한 문구 표시 — 특정인을 지적하지 않음 (PRD 8절)
- **동작**: 넛지는 알아서 5~10초 후 자동으로 사라짐, 사용자가 무시해도 세션은 계속됨(강제 종료 없음)
- **완료 기준**: 넛지가 판정처럼 느껴지지 않는지 초기 사용자 인터뷰에서 확인 필요 (열린 질문으로 남김, 3절 참고)

### 2.6 2단계: 의미 레벨 감지 + 리프레이징 제안

- **선행 조건**: 두 기기 구조 채택(2026-07-15)으로 완화됨 — 각자 스트림이 이미 분리되어 있으므로 "겹치지 않는 구간"을 기다릴 필요 없이 상시 동작 (기술 스펙 4절 참고)
- **동작**: 각 기기 오디오를 STT로 텍스트 변환 → 비난/경멸/방어(Gottman Four Horsemen 기반) 표현 감지 → 감지되면 리프레이밍 제안 카드 표시
- **완료 기준**: 제안은 참고용 카드로만 표시, 자동으로 대신 말해주거나 강제하지 않음
- **명시적 비목표**: 항상 텍스트 전문을 저장/노출하지 않음 (제안 카드만 노출, 전체 스크립트는 요약 화면에도 원문 그대로 노출하지 않는 방향을 기본값으로 — 프라이버시 우선, 필요시 사용자가 명시적으로 "대화 내용 보기" 선택했을 때만 노출은 열린 논의 사항)

### 2.7 세션 종료

- **트리거**: 메인 화면의 단일 버튼("끝내기") 탭
- **동작**: 마이크 즉시 비활성화 → 2.8 요약 화면으로 이동
- **완료 기준**: 탭 즉시 마이크 표시가 꺼짐 (신뢰의 핵심 — "끝냈다고 했는데 아직 듣고 있다"는 느낌이 있으면 신뢰 붕괴)

### 2.8 세션 요약 화면

- **정의**: 세션 동안의 감정 온도(격화 스코어) 그래프, 노출됐던 리프레이징 제안 목록을 양쪽이 함께 볼 수 있게 표시
- **동작**: 요약 확인 후 "저장" 또는 "삭제" 중 선택 (기본값은 삭제 — opt-in 저장)
- **완료 기준**: 추가 입력(예: 오늘 기분 평가, 퀴즈 등)을 요구하지 않음 — 보기만 하고 끝낼 수 있어야 함

### 2.9 데이터 처리 정책

- **원본 오디오**: 세션 종료 시 즉시 폐기 (서버에 영구 저장하지 않음)
- **텍스트 트랜스크립트/제안 로그**: 세션 단위로 임시 보관, 사용자가 "저장" 선택 시에만 유지, 기본값은 세션 종료와 함께 삭제
- **완료 기준**: 개발 착수 전 실제 저장/폐기 로직에 대한 QA 체크리스트 별도 작성 필요 (본 문서 6절 오픈 이슈로 등록)

## 4. 유저 스토리 (요약)

- 우리가 다투다가 격해지면, 누구든 한 명이 "시작" 버튼만 눌러서 바로 개입을 받을 수 있다.
- 대화 중 격해지면, 나와 상대 둘 다 같은 중립적 메시지를 보게 된다 (한쪽만 혼나는 느낌이 없다).
- 비난하는 말을 했을 때, 다르게 말할 수 있는 방법을 참고용으로 볼 수 있다 (강요받지 않는다).
- 세션이 끝나면 무슨 일이 있었는지 요약으로 볼 수 있고, 원하면 지울 수 있다.

## 5. 명시적 비목표 (재확인)

PRD 4-1절 "안 만들 것" 표와 동일. 본 문서에서 별도로 반복하지 않고 참조만 함. 새로운 비목표가 논의되면 PRD와 본 문서 양쪽에 동시 반영.

## 6. 오픈 이슈 (개발 착수 전 결정 필요)

1. 격화 스코어 임계치의 초기값을 어떻게 정할지 — 소규모 실제 대화 샘플로 캘리브레이션 필요
2. 세션 요약 화면에 원문 텍스트를 얼마나 노출할지 (프라이버시 vs 유용성 트레이드오프) — 미결
3. 위기 신호(가정폭력 등) 감지 기준과 전문기관 연결 분기 로직 — 별도 문서로 분리 예정, MVP 개발과 병행 착수 권장
4. 페어링 링크의 보안(임의의 제3자가 링크를 알게 됐을 때의 위험) 처리 방식
5. 넛지가 "판정처럼 느껴지는지"를 실제로 확인할 초기 사용자 테스트 설계
6. **(신규, 2026-07-15)** B가 "참여"를 누르는 것을 A가 어떻게 알리는가 — 말로 알리는 것만으로 충분한지, 화면에 공유 가능한 QR/짧은 코드라도 필요한지
7. **(신규)** 두 사람이 물리적으로 가까이 있을 때 마이크 크로스토크(상대 목소리가 내 기기에도 잡히는 현상)를 실제로 얼마나 방해하는지 — 프로토타입 초기 실측 필요 (기술 스펙 7절 한계 1번)
8. **(신규)** 기기 간 타임스탬프 동기화 방식 확정 — 서버 수신 시각 기준으로 충분한지, 별도 clock sync가 필요한지
